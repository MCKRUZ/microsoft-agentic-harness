The Story

This session laid the first concrete foundation of the Microsoft Agentic Harness — a template project for building AI agent applications on Microsoft Agent Framework. Starting from an empty repo with only planning documents, we built the complete exception hierarchy for the Application.Common layer.

The work drew from two source codebases: the ApplicationTemplate (the C# Clean Architecture reference implementation) provided 7 enterprise exception types, and Claude Code's leaked source (TypeScript) revealed 6 additional exception patterns unique to agentic systems. The session followed a deliberate design-then-implement cadence: analyze, propose, get user approval, implement, review, fix.

Chapter 1: Scaffolding

Project Structure & Build Infrastructure

Created the initial directory structure following Clean Architecture conventions from the ApplicationTemplate. The src/Content/Application/Application.Common/ path mirrors the reference exactly.

The .csproj targets net10.0, enables nullable reference types, and sets RootNamespace to Application.Common. FluentValidation is the sole dependency — pinned to 11.11.0 (originally wildcarded to 11.*, fixed in the review pass).

Chapter 2: Enterprise Exception Types (from ApplicationTemplate)

7 exceptions ported from C# reference implementation

Each of the ApplicationTemplate's 7 exception types was analyzed for structural issues, then ported with a uniform constructor pattern and full XML documentation. The originals had inconsistencies: some had inner exception constructors, some didn't; one used int? for keys; another had [JsonConstructor] on ambiguous overloads.

  • Exceptions/ExceptionTypes/BadRequestException.cs HTTP 400
  • Exceptions/ExceptionTypes/ValidationException.cs FluentValidation
  • Exceptions/ExceptionTypes/EntityNotFoundException.cs HTTP 404
  • Exceptions/ExceptionTypes/ConfigurationNotFoundException.cs Config
  • Exceptions/ExceptionTypes/ForbiddenAccessException.cs HTTP 403
  • Exceptions/ExceptionTypes/DatabaseInteractionException.cs Data Access
  • Exceptions/ExceptionTypes/NoContentException.cs HTTP 204
Decision: Port all 7, not just what the harness needs today

Initially proposed skipping DatabaseInteractionException and ForbiddenAccessException since the harness POC has no database or auth. The user corrected this: "this is meant to be a template of sorts — there are going to be a lot of things that may not make sense right away, but would be good for future behavior." All 7 were ported.

Decision: Keep full XML documentation

Initially proposed trimming XML docs for conciseness. The user pushed back: templates are picked up by people who didn't write them. XML docs power IntelliSense, show correct usage patterns, and explain when to use one exception vs another. Docs stayed comprehensive.

Chapter 3: Agentic Exception Types (from Claude Code patterns)

6 new exceptions designed for agentic systems

These exceptions address failure modes unique to AI agent applications — none existed in the ApplicationTemplate because it wasn't agent-focused yet. Each was designed by studying Claude Code's error handling patterns and translating them to C# idioms.

  • Exceptions/ExceptionTypes/ToolExecutionException.cs Agent
  • Exceptions/ExceptionTypes/SkillNotFoundException.cs Agent
  • Exceptions/ExceptionTypes/AgentExecutionException.cs Agent
  • Exceptions/ExceptionTypes/McpConnectionException.cs Agent
  • Exceptions/ExceptionTypes/ContentSafetyException.cs Agent
  • Exceptions/ExceptionTypes/ContextBudgetExceededException.cs Agent

Each carries domain-specific properties (e.g., ToolName, ServerName, TokenLimit) that enable the agent orchestration loop to make informed recovery decisions rather than just logging generic error strings.

Chapter 4: Code Review & Fixes

Automated review caught 11 issues; all resolved

HIGH fixes

H2: Bool sentinel hack in ConfigurationNotFoundException

- public ConfigurationNotFoundException(string section, bool sectionMissing)
+ public static ConfigurationNotFoundException ForSection(string section)
+ => new(...) { Section = section };

The bool sectionMissing parameter existed only to disambiguate from (string message). Replaced with a static factory method — cleaner API, impossible to misuse.

H3: Mutable Errors dictionary in ValidationException

- public IDictionary<string, string[]> Errors { get; }
+ public IReadOnlyDictionary<string, string[]> Errors { get; }

Violated the project's immutability-first rule. All constructors now produce ReadOnlyDictionary instances.

MEDIUM fixes

M1: Added XML warning on NoContentException that Result<T> is preferred over exception-based flow control.

M2: Pinned FluentValidation from 11.* to 11.11.0.

M3: Changed ContextBudgetExceededException token properties from int to long for compatibility with Semantic Kernel's token counting.

M4: Added ArgumentException.ThrowIfNullOrWhiteSpace guards on all type-specific constructors across all 13 exceptions. ContextBudgetExceededException also got ArgumentOutOfRangeException.ThrowIfNegative on token values.

M5: Added sealed to all 13 exception classes — these are leaf types in the hierarchy.

LOW fix

L2: Created ApplicationExceptionBase abstract class so global exception handlers can catch (ApplicationExceptionBase) instead of listing every type. All 13 exceptions now inherit from it instead of System.Exception.

Uniform Pattern Established

Every exception in this hierarchy now follows the same contract:

public sealed class XException : ApplicationExceptionBase
{
// Domain-specific properties (if any)
public string? PropertyName { get; }
// 1. Parameterless — sensible default message
public XException() : base("Default.") { }
// 2. Custom message
public XException(string message) : base(message) { }
// 3. Wrap with inner exception
public XException(string message, Exception inner) : base(message, inner) { }
// 4. Type-specific (with argument guards)
public XException(string name, string? detail, Exception? inner = null)
: base(FormatMessage(name, detail), inner)
{
ArgumentException.ThrowIfNullOrWhiteSpace(name);
PropertyName = name;
}
}

Decisions & Rules Established

Rule: Never write code without user approval

Saved as project memory. Every change must be presented with rationale before implementation.

Rule: Build for the future (template mindset)

Include components that future consumers will need, not just what the harness uses today.

Rule: Full XML documentation on all public types

This is a template. Docs are teaching material, not bloat.

Rule: Post-change review cadence

/code-review + /review-changes per folder. /simplify per layer.
