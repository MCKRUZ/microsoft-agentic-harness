Presentation.Common — Composition Root

Eliminating the god object, building the composition root, and wiring every layer of the agentic harness into a single GetServices() call.

Project: Microsoft Agentic Harness Date: April 7, 2026 Phases: A (Refactor) + B (Build) Files: 28+ modified/created

1 The Story

The Problem: AppConfig as God Object

Every DependencyInjection.cs across the entire solution accepted AppConfig as a parameter — a root object containing all configuration for every layer. Services that only needed logging config received the full Azure, AI, HTTP, and cache configuration trees. Runtime consumers injected IOptionsMonitor<AppConfig> when they only read a single subsection.

This created invisible coupling: any change to any config section potentially affected every service in the solution. It also made testing harder — tests had to construct the entire AppConfig graph just to test one component.

The Solution: Two-Phase Surgery

Phase A Narrow the parameters. Every DI method was refactored to accept only the config subsection it actually needs — or nothing at all, resolving from IOptionsMonitor<T> at runtime. Nine runtime consumers were changed from IOptionsMonitor<AppConfig> to their specific config type.

Phase B Build the composition root. A new Presentation.Common project became the single place that knows about AppConfig as a whole. It binds all config sections, registers all layers, and builds the OTel pipeline — so no other project needs to see the god object.

Why order matters

Each project's DependencyInjection.cs registers its ITelemetryConfigurator as a singleton. The OTel pipeline in OpenTelemetryServiceCollectionExtensions resolves all configurators from DI and applies them in Order sequence. If OTel were registered before the project dependencies, the configurators would not exist yet.

The IDeferredTracerProviderBuilder / IDeferredMeterProviderBuilder pattern avoids the BuildServiceProvider() anti-pattern — configurators are resolved from the real IServiceProvider at pipeline-build time, not from a throwaway intermediate provider.

5 Code Review Findings & Fixes

CRITICAL Sensitive Telemetry Hardcoded

Finding: The EnableOTelDiagnosticsSensitive AppContext switch was hardcoded to true, meaning all GenAI prompt and completion content would be recorded in traces regardless of environment.

Fix: Made it config-driven via appConfig.Observability.EnableSensitiveTelemetry, defaulting to false. Production requires explicit opt-in.

// Before (DANGEROUS)
AppContext.SetSwitch("...EnableOTelDiagnosticsSensitive", true);

// After (config-driven, default false)
AppContext.SetSwitch("...EnableOTelDiagnosticsSensitive",
    appConfig.Observability.EnableSensitiveTelemetry);

CRITICAL Missing Secret Storage Warnings

Finding: RedisClientConfig.Secret and EntraCredentialConfig.ClientSecret had no XML doc warnings about secret storage requirements.

Fix: Added explicit warnings: "This value must NEVER be stored in appsettings.json or any file committed to source control. Use User Secrets (development) or Azure Key Vault (production)."

HIGH BuildServiceProvider() Anti-Pattern

Finding: Original OTel registration called services.BuildServiceProvider() to resolve ITelemetryConfigurator instances, creating a throwaway DI container and duplicate singletons.

Fix: Used IDeferredTracerProviderBuilder and IDeferredMeterProviderBuilder to defer configurator resolution to the real IServiceProvider at pipeline-build time.

HIGH File Size Violation

Finding: IServiceCollectionExtensions.cs exceeded line limits when health checks were inline.

Fix: Extracted HealthCheckServiceCollectionExtensions.cs as a dedicated file with AddCustomHealthChecks, AddAzureHealthChecks, AddCacheHealthChecks, and AddHealthCheckPublishers methods.

HIGH Path Validation Missing

Finding: PropagateAppConfigChanges did not validate file paths, risking path traversal.

Fix: Added File.Exists check for source, ArgumentNullException.ThrowIfNull for both parameters, and FileNotFoundException when source is missing.

HIGH AzureConfig One-File-Per-Class

Finding: All 7 Azure config classes were in a single AzureConfig.cs file.

Fix: Split into 7 individual files: AzureConfig.cs, ApplicationInsightsConfig.cs, AzureDatabaseConfig.cs, AzureADB2CConfig.cs, KeyVaultConfig.cs, GraphApiConfig.cs, EntraCredentialConfig.cs.

7 Key Architecture Decisions

1. Centralized Config Binding

RegisterConfigSections binds all 11 config classes as IOptions<T> in one place. No other project calls services.Configure<T>. This guarantees every subsection is available for runtime resolution, and there is exactly one place to add new sections.

2. Narrow or Nothing

DI methods either take nothing (resolving config at runtime from IOptionsMonitor<T>) or their specific config subsection when they need it at registration time. The root AppConfig object is only referenced inside Presentation.Common.

3. ITelemetryConfigurator Extensibility

Each project registers its own ITelemetryConfigurator with an Order property. The OTel pipeline discovers all of them from DI and applies them in sequence. Adding a new telemetry source means implementing one interface — no changes to the pipeline code.

4. Registration Order = Execution Order

Project dependencies are registered before OTel so configurators exist when the pipeline is built. MediatR behaviors are registered in execution order (outermost first). This ordering is documented in XML comments and enforced by the composition root.

10 What Program.cs Looks Like Now

// The entire composition root reduces to this:

var builder = WebApplication.CreateBuilder(args);

// One call registers everything: config, caching, health checks,
// all layer dependencies, and the OTel pipeline.
builder.Services.GetServices();

// Web-specific registrations (Kestrel, versioning, Swagger, CORS)
builder.Services.AddPresentationCommonDependencies(httpConfig);

// Auth (optional — skips B2C when not configured)
builder.Services.AddAuthDependencies(azureConfig);

var app = builder.Build();

// Middleware in correct order
app.UseAllMiddleware();
app.UseAuthentication();
app.UseAuthorization();

// Health check UI
app.AddHealthCheckEndpoint("/api");

app.MapControllers();
app.Run();
