The Conversation Loop

You know that moment when an AI assistant pauses, calls a tool, gets results, and then keeps going? That is not one step — it is an entire loop running behind the scenes, and every iteration goes through a gauntlet of checks.

Think of it like a chess game with a coach. The agent (the player) studies the board and picks a move. The coach (the harness) checks the move is legal. Then the opponent (reality, via a tool call) responds. The game continues until checkmate — or the clock runs out.

The Loop Secret

The conversation is not one big API call. It is a loop — each turn goes through validation, safety, and the LLM, then checks: “Did the AI ask for a tool, or is it done?” Tool call means keep looping. No tool call means we are finished.

The Gauntlet: 6 Pipeline Behaviors

Before the handler ever touches your message, it must survive a gauntlet of six checkpoints. These are called pipeline behaviors, and they run in this exact order every single time.

1
Validation

Checks that your request has all required fields and nothing looks malformed. Catches bad data before it wastes anyone’s time.

2
Caching

Checks if this exact request was already answered recently. If so, returns the cached result instantly — no need to bother the LLM.

3
Content Safety

Scans the message for harmful, toxic, or policy-violating content. Blocks dangerous requests before they ever reach the AI.

4
Tool Permission

Verifies this agent is actually allowed to use the tools it might call. An agent cannot access the file system if it was not granted that permission.

5
Performance Logging

Starts a timer and records telemetry so you can see exactly how long each step takes. Essential for diagnosing slow responses.

6
Handler (The Actual Work)

Finally! The request reaches the handler, which calls the LLM, processes tool calls, and returns the result. Everything before this was just making sure we are safe to proceed.

This pattern is called CQRS with MediatR. Every request is a “command” that gets routed through behaviors before reaching a handler. The order matters — validation first, because there is no point checking content safety on a malformed request.

ExecuteAgentTurn — One Round

This is the single most important piece of code in the harness. It handles one turn — send messages to the LLM, check if it asked for tools, and report back whether we are done.

CODE 

public async Task<AgentTurnResult> Handle(ExecuteAgentTurnCommand request, CancellationToken cancellationToken)
{
    using var turnActivity = AgentActivitySource.StartTurn(
        request.Context.AgentName, request.TurnIndex);

    _logger.LogDebug("Executing turn {TurnIndex} for agent {AgentName}",
        request.TurnIndex, request.Context.AgentName);

    try
    {
        var response = await request.Agent.InvokeAsync(
            request.Messages, request.Context.AgentOptions, cancellationToken);

        var updatedMessages = request.Messages.ToList();
        updatedMessages.Add(response);

        var hasToolCalls = response.Contents.OfType<FunctionCallContent>().Any();
        var isComplete = !hasToolCalls;

        OrchestrationMetrics.ToolCalls.Add(
            hasToolCalls ? response.Contents.OfType<FunctionCallContent>().Count() : 0,
            new KeyValuePair<string, object?>("agent.name", request.Context.AgentName));

        return new AgentTurnResult(updatedMessages, isComplete, request.TurnIndex);
    }
    catch (Exception ex)
    {
        _logger.LogError(ex, "Turn {TurnIndex} failed for {AgentName}",
            request.TurnIndex, request.Context.AgentName);
        throw new AgentExecutionException(request.Context.AgentName, request.TurnIndex, ex);
    }
}
PLAIN ENGLISH

This function handles one turn. It receives the command and a way to cancel if needed.

 

Start a tracing span so we can measure how long this turn takes.

 

Write a log entry: “Starting turn #X for agent Y.”

 

Now try the real work:

 

Send all the messages so far to the LLM and wait for its response. The cancellation token lets us bail out if the user disconnects.

 

Take the existing message history, copy it, and append the new response from the LLM to the end.

 

Check: did the LLM ask to use any tools? Look for FunctionCallContent objects in the response.

If there are NO tool calls, the turn is complete. If there ARE tool calls, we need another turn.

 

Record metrics: how many tool calls were made this turn, tagged with the agent name. This feeds the performance dashboards.

 

Return the result: updated messages, whether we are done, and which turn number this was.

 

If anything goes wrong, log the error with details about which turn and which agent failed.

Then wrap it in a custom exception so the calling code knows exactly what broke and where.

 

RunConversation — The Full Loop

The previous screen showed one turn. This is the loop that chains turns together — keep going until the agent finishes or we hit the turn limit.

CODE 

while (turnCount < request.MaxTurns)
{
    turnCount++;
    var turnResult = await _mediator.Send(
        new ExecuteAgentTurnCommand(agent, messages, context, turnCount),
        cancellationToken);

    messages = turnResult.UpdatedMessages;

    if (turnResult.IsComplete)
    {
        _logger.LogInformation("Conversation completed after {TurnCount} turns", turnCount);
        break;
    }
}
PLAIN ENGLISH

Keep looping as long as we have not hit the maximum number of turns.

 

Bump the turn counter and send an ExecuteAgentTurn command through MediatR. This triggers the entire pipeline — validation, safety, permission checks — before reaching the handler we saw on the previous screen.

 

 

Update our message history with whatever the LLM said (and any tool results).

 

Did the LLM finish without asking for any tools?

 

Yes — log that we are done and how many turns it took.

Break out of the loop. The conversation is over.

 

💡
Two Ways the Loop Ends

Either the LLM decides it is done (IsComplete = true, no tool calls) or the turn counter hits MaxTurns and the while loop simply stops. This is a safety net — without MaxTurns, a confused agent could loop forever.

Going Multi-Agent: The Orchestrator

One agent running a conversation loop is powerful. But what if the task is too big for one agent? The orchestrator breaks it into subtasks, hands each one to a specialist sub-agent, and runs them all at once.

CODE 

// Phase 1: Create orchestrator and get task decomposition
var agentCatalog = BuildAgentCatalog(request.AvailableAgents);

var orchestratorContext = await _contextFactory.CreateAsync(
    request.OrchestratorName, cancellationToken);
var orchestrator = _agentFactory.CreateAgent(orchestratorContext.Definition);

// Phase 2: Decompose and delegate
var decomposition = await GetTaskDecomposition(
    orchestrator, request.Task, agentCatalog, cancellationToken);

// Phase 3: Run sub-agent conversations in parallel
var subtaskResults = await Task.WhenAll(
    decomposition.Subtasks.Select(subtask =>
        RunSubagentConversation(subtask, cancellationToken)));
PLAIN ENGLISH

Phase 1: Build a catalog of all available agents so the orchestrator knows who can do what.

 

Set up the orchestrator agent — load its configuration and context.

 

Create the orchestrator agent from its definition (using the same factory pattern from Module 2).

 

Phase 2: Ask the orchestrator to analyze the task and break it into smaller subtasks, deciding which agent should handle each one.

 

 

Phase 3: Run ALL the sub-agent conversations at the same time using Task.WhenAll.

For each subtask, spin up a full conversation loop (just like RunConversation) and run them all in parallel.
